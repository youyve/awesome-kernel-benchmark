# Awesome Kernel Benchmark [![Awesome](https://awesome.re/badge.svg)](https://awesome.re)

> A curated, **scientifically-organized** catalog of benchmarks for GPU / NPU / accelerator **kernel** generation, optimization, and translation — the workloads that LLM-driven **kernel agents** are evaluated on.

Companion to [**awesome-kernel-agent**](https://github.com/youyve/awesome-kernel-agent) (the *systems & agents*); this list is the *benchmarks*. The two are deliberately split: an agent paper that reports "we optimize PolyBench-ACC / NPB / XSBench kernels" is using a **substrate suite**, not a purpose-built agent benchmark — and the field's benchmark landscape is large enough to deserve its own home.

## Why two layers

Kernel agents are evaluated in **two distinct layers**. Conflating them inflates counts and breaks comparability.

1. **Purpose-built agent benchmarks** — harnesses *designed* to score LLM kernel generation (correctness + speedup vs a reference): KernelBench, TritonBench, CUDABench, SOL-ExecBench, FlashInfer-Bench, … These are the headline numbers.
2. **Foundational / substrate suites** — pre-LLM HPC / GPU / DL benchmark suites that agents are evaluated *on*, fine-tuned *against*, or asked to *optimize / translate*: PolyBench, NPB, XSBench, Rodinia, HeCBench, DeepBench, SuiteSparse, … Many agents report **only** on these.

The **[agent ↔ substrate map](#agent--substrate-map)** links the two: for any suite, which agents already run on it (so results are comparable) and which ship re-runnable solution kernels (so they are auditable).

## How this catalog is organized

The design follows established benchmark-taxonomy practice — Berkeley's *13 computational motifs* ([View from Berkeley](https://www2.eecs.berkeley.edu/Pubs/TechRpts/2006/EECS-2006-183.pdf)), the abstraction-level / granularity axis used by SPEC, MLPerf and code-LLM surveys, and the **YAML-single-source-of-truth → generated tables** pattern of the [Herten et al. HPC benchmark survey](https://fzj-jsc.github.io/benchmark-survey/) (IJHPCA 2026) and [awesome-selfhosted-data](https://github.com/awesome-selfhosted/awesome-selfhosted-data).

- **Single source of truth:** [`data/benchmarks.yaml`](data/benchmarks.yaml). Every table below is generated by [`scripts/generate.py`](scripts/generate.py) — **do not hand-edit the tables.** A machine-readable [`data/benchmarks.json`](data/benchmarks.json) is exported for tooling.
- **Multi-tag, faceted classification** (not single-bucket). Each entry carries orthogonal facets — see [`SCHEMA.md`](SCHEMA.md) for the data dictionary and controlled vocabularies:
  - **layer** — agent-benchmark · substrate-suite · dataset · tooling
  - **abstraction** *(primary axis)* — micro · kernel · fused-op · operator · proxy-app · application · suite · dataset · harness
  - **motifs** *(secondary axis)* — the Berkeley 13 (dense-LA, sparse-LA, spectral, n-body, structured-grid, unstructured-grid, monte-carlo, graph-traversal, dynamic-programming, combinational-logic, branch-and-bound, graphical-models, finite-state-machine) + DL extensions (attention, reduction-scan, elementwise, data-movement)
  - **bottleneck** — compute · bandwidth · latency · communication (roofline regime)
  - **programming model · hardware · precision · domain · agent-role · verify · ships-kernels · status**
- **Coverage matrix** (motif × family) surfaces under-served motifs at a glance.
- **Column legend:** **Verify** = built-in correctness oracle (✅ reference/checksum/residual · ◐ partial · — none) — this is what makes a suite *anti-gaming*. **Ships** = re-runnable reference/solution kernels (✅ · ◐ harness/vendor-backed · — none) — ✅ suites double as a human baseline *and* an audit target.

## Methodology & selection criteria

**Inclusion scope.** A benchmark belongs here if a kernel agent could plausibly be *evaluated on*, *fine-tuned against*, or *asked to optimize/translate* it. That spans purpose-built agent harnesses, classic GPU/HPC suites, DL operator micro-benchmarks, sparse/graph/stencil suites, roofline microbenchmarks, compiler test suites, and the datasets/tooling around them.

**Abstraction levels** (the primary axis), defined to keep tagging consistent:
`micro` (single arithmetic/bandwidth probe) · `kernel` (one GPU kernel) · `fused-op` (a fused operator sequence) · `operator` (a full DL operator / layer) · `proxy-app` (a mini-app standing in for a real application) · `application` (full end-to-end model/app) · `suite` (a curated collection) · `dataset` (inputs only, no kernels) · `harness` (measurement framework, no fixed kernels).

**Multi-motif is allowed and expected** — attention = dense-LA + reduction-scan/softmax; SpMV = sparse-LA + graph-traversal. Motif assignments follow the suite's own paper where it self-classifies (OpenDwarfs, Rodinia), otherwise they are an editorial call per the Berkeley definitions.

**Provenance & dedup.** Suites that repackage others (SPEC ACCEL ⊂ Parboil/Rodinia/NPB; the many KernelBench derivatives) are marked rather than double-counted. Kernels that recur across suites (SpMV, SGEMM, BFS, FFT, stencil) are noted, not duplicated.

**Liveness.** A `status` field tracks active/maintained/archived/retired — catalogs rot (SPEC ACCEL was silently retired; Papers-with-Code shut down), so the source of truth is durable git + YAML, not a hosted DB.
