## ① 我该用什么评测我的 agent？

按你要*证明*的事情来选：

- **"我的数字和已有工作可比"** → [KernelBench](https://github.com/ScalingIntelligence/KernelBench)——事实标准，人人都报。但要清楚它的缺陷（见记分卡），绝不能作为唯一证据。
- **"我的加速比不是作弊产物"** → [robust-kbench](https://github.com/SakanaAI/robust-kbench)——反作弊过滤、forward+backward、1e-5 容差。
- **"我的 kernel 在真实生产形状上正确"** → [BackendBench](https://github.com/meta-pytorch/BackendBench)——PyTorch OpInfo 边界用例 + TorchBench 生产 trace；kernel 以 pip 后端形式可部署。
- **"我的 kernel 在绝对意义上快，不只是比 eager 快"** → [SOL-ExecBench](https://research.nvidia.com/benchmarks/sol-execbench) / [CUDABench](https://github.com/CUDA-Bench/CUDABench)——天花板相对评分（注意：两者都用 datasheet 峰值，比实测 ceiling 松 8–15%）。
- **"我的优化在真实 serving 栈里站得住"** → [ISO-Bench](https://github.com/Lossfunk/ISO-Bench)（vLLM/SGLang 真实合并 PR 任务）/ [FlashInfer-Bench](https://github.com/flashinfer-ai/flashinfer-bench)。
- **"我的结果对算力预算诚实"** → [GEAK](https://github.com/AMD-AGI/GEAK)——唯一把 sequential@k 与 parallel@k 分开报的。
- **专做 Triton** → [TritonBench](https://github.com/thunlp/TritonBench)；**ROCm/AMD** → GEAK、[AgentKernelArena](https://github.com/AMD-AGI/AgentKernelArena)、[NPUEval](https://github.com/AMDResearch/NPUEval)；**昇腾 NPU** → [MultiKernelBench](https://github.com/wzzll123/MultiKernelBench)、[AscendKernelBench](https://github.com/mindspore-ai/akg)、[NPUKernelBench](https://openi.pcl.ac.cn/PCL-Benchmark/NPUKernelBench)；**CUDA 正确性（带隐藏测试）** → [ComputeEval](https://github.com/NVIDIA/compute-eval)。

**一篇严肃论文的诚实答案是组合拳：** 一个可比性锚点（KernelBench）+ 一个加固 oracle（robust-kbench 或 BackendBench）+ 一个绝对指标（SOL-ExecBench 风格）+ 显式预算上报。没有任何单一 benchmark 同时覆盖这四样——这个空缺在下面的记分卡里一眼可见。

## ② 各 benchmark 到底怎么测？——方法学记分卡

同一个 agent 在两个 benchmark 上可以得到天差地别的数字，因为分数是五个设计选择的乘积：**任务集**、**正确性 oracle**、**计时方法学**、**基线**、**预算核算**。这张记分卡对区分"可信数字"与"虚高数字"的那些选择逐项评级——**只依据一手证据**（harness 源码/论文）评级，绝不采信 README 的自我宣称。

## 附录 B——这份列表是怎么构建的

- **数据层：** [`data/benchmarks.yaml`](data/benchmarks.yaml)（每个 benchmark 一条，多标签 facet：layer、abstraction、motif、bottleneck、hardware、verify、ships、status——完整词表见 [`SCHEMA.md`](SCHEMA.md)）+ [`data/scorecard.yaml`](data/scorecard.yaml)（方法学评级，附证据链接）。[`scripts/generate.py`](scripts/generate.py) 负责校验与渲染；`--check` 跑在 CI 里。
- **收录范围：** 一切 kernel agent 可能被评测于其上、针对其微调、或被要求优化/翻译的东西。
- **出处与去重：** 重新打包其他套件的（SPEC ACCEL ⊂ Parboil/Rodinia/NPB；KernelBench 衍生品）只标注不重复计数。`status` 字段追踪存活状态——目录是会腐烂的。
- **设计血统：** Berkeley [13 计算 motif](https://www2.eecs.berkeley.edu/Pubs/TechRpts/2006/EECS-2006-183.pdf)、[Herten 等的 HPC benchmark 综述](https://fzj-jsc.github.io/benchmark-survey/)的 YAML→表格模式、以及本列表首创的方法学记分卡层。

## 术语表

| 术语 | 白话含义 |
|:---|:---|
| **kernel agent** | 用 LLM 写/优化 GPU kernel 的系统，通常是"生成→profile→精化"循环 |
| **substrate / 任务源** | 被 agent 当原材料用的经典 benchmark 套件——本身不是为给 agent 打分设计的 |
| **正确性 oracle** | 判定"这个 kernel 的输出对不对"的机制——它的严格度决定作弊 kernel 能否得分 |
| **motif** | 基本计算模式（稠密线代、stencil、图遍历、attention……）；描述覆盖度的词汇 |
| **polyhedral（套件）** | 规则嵌套循环数组运算的 kernel（GEMM 类）——最易分析、校验与变换 |
| **proxy/mini-app** | 真实科学应用的几千行替身（如 XSBench ≈ 反应堆代码的热循环） |
| **verify ✅ / ships ✅** | 套件自带正确性检查 / 附可复跑参考 kernel（反作弊、可审计） |
| **roofline / SOL** | kernel 至多能达到的硬件天花板（算力或带宽）；NVIDIA 行话叫"speed-of-light" |
| **pass@k / 预算** | 给 k 次尝试时的得分；不声明预算，best-of-N 和单次生成的结果不可比 |
